# AWS S3

## Access control — bucket policies vs ACLs

Modern S3 uses **bucket policies + IAM** for access control. ACLs are legacy. The 2023 default for new buckets is `BucketOwnerEnforced` Object Ownership, which **disables ACLs entirely** and makes the bucket owner the sole controller of objects.

| Ownership setting | ACLs | Recommended |
|---|---|---|
| `BucketOwnerEnforced` | Disabled | **Yes** — default for new buckets since 2023 |
| `BucketOwnerPreferred` | Enabled, owner takes new objects | Migration step only |
| `ObjectWriter` | Enabled, uploader retains ownership | Legacy; avoid for new buckets |

Migration path: existing `ObjectWriter` buckets → `BucketOwnerPreferred` (rewrite ownership on new uploads) → `BucketOwnerEnforced` (ACLs off). Audit upload pipelines first; some SDKs default to setting ACLs which fail under enforced ownership.

## Block Public Access (BPA)

Both **account-level** and **bucket-level**. All four switches should be on by default:

- `BlockPublicAcls` — reject ACL-based public grants on PUT
- `IgnorePublicAcls` — ignore existing ACL-based public grants on GET
- `BlockPublicPolicy` — reject bucket policies that grant public access
- `RestrictPublicBuckets` — restrict access to public buckets to AWS service principals only

Account-level BPA overrides per-bucket settings. Disable a bucket's BPA only with explicit, audited justification (e.g., a website-hosting bucket with `s3:GetObject` for `*`).

## Encryption

| Mode | Key management | Use case |
|---|---|---|
| SSE-S3 (default) | AWS-managed AES-256 | Baseline; encrypted-at-rest by default since 2023 |
| SSE-KMS | Customer-managed KMS key | Audit + key-policy control; charges per KMS request |
| DSSE-KMS | Two layers of encryption with KMS | High-compliance only; FIPS 140-2 dual-layer |
| SSE-C | Customer-supplied key per request | Niche; key management is your problem |

SSE-KMS adds latency + cost per request. **S3 Bucket Keys** (`BucketKeyEnabled: true`) reduce KMS API calls by ~99% by deriving per-object data keys from a single bucket-level KMS key reference — enable for any SSE-KMS bucket.

## Versioning + lifecycle

- **Versioning** is required for replication, MFA Delete, and Object Lock. Once enabled, can only be suspended (not disabled).
- **Lifecycle** rules transition objects between storage classes (Standard → Standard-IA → Glacier Instant → Glacier Flexible → Deep Archive) and expire versions. Per-prefix or per-tag filters.
- **Intelligent-Tiering** is the "just figure it out" alternative — small monitoring fee per object, automatic tier transitions based on access pattern. Default for unknown-access-pattern workloads.

## Object Lock

WORM (write-once-read-many) for compliance. Two modes:

- **Governance** — only principals with `s3:BypassGovernanceRetention` can override.
- **Compliance** — no one can override, including root. Set retention period carefully.

Object Lock requires Versioning. Cannot be disabled once enabled on a bucket. Default retention can be set bucket-wide.

## Replication (CRR / SRR)

- **CRR** = Cross-Region Replication
- **SRR** = Same-Region Replication
- Requires Versioning on source + destination.
- IAM role for replication needs `s3:GetReplicationConfiguration`, `s3:ListBucket` on source and `s3:ReplicateObject`, `s3:ReplicateDelete`, `s3:ReplicateTags` on destination.
- KMS-encrypted source needs `kms:Decrypt` on source key + `kms:Encrypt` on destination key in the replication role.
- **RTC** (Replication Time Control) — 15-minute SLA for replication; adds cost; required for some compliance regimes.

## Presigned URLs

Time-limited URLs that grant temporary access using the signing principal's credentials. Max expiry is the credential lifetime — for STS-derived credentials, that's the session length (default 1h, max 12h for chained roles). For long-lived IAM-user keys, the expiry can be up to 7 days.

The URL inherits the signing principal's permissions. **Do not** sign with admin credentials and share publicly; sign with a narrowly-scoped role and rotate often.

## Multipart upload

- Mandatory for objects > 5 GiB.
- Recommended for objects > 100 MiB.
- Incomplete multipart uploads persist and incur storage cost. **Add a lifecycle rule** to abort incomplete multipart uploads after 7 days. Easy to forget; common cost leak.

## Common pitfalls

- **`Action: "s3:*", Resource: "*"`** — over-broad. Scope to specific buckets and prefixes (`arn:aws:s3:::bucket/prefix/*`).
- **Missing `s3:ListBucket` on bucket ARN** when granting object actions — `aws s3 ls` and recursive operations fail without it. Bucket ARN is `arn:aws:s3:::bucket` (no trailing slash, no path); object ARN is `arn:aws:s3:::bucket/key`.
- **Cross-account access without bucket policy** — IAM allow on the caller is necessary but not sufficient. The bucket policy must explicitly allow the cross-account principal.
- **KMS-encrypted bucket cross-account** — the caller's IAM role needs `kms:Decrypt` on the bucket's KMS key, AND the KMS key policy must allow the role.
- **BPA disabled at account level** — bucket-level BPA is overridden silently.
- **Incomplete multipart uploads accumulating** — invisible in console object count; visible only via `aws s3api list-multipart-uploads`. Set lifecycle.
- **VPC endpoint policy too narrow** — restricts which buckets a VPC can reach. `Action: "*", Resource: "*"` is the open default; tighten per bucket only after auditing.
- **Bucket name in URL vs path style** — virtual-hosted-style (`bucket.s3.region.amazonaws.com`) is current; path-style (`s3.region.amazonaws.com/bucket`) is being deprecated for new buckets.
