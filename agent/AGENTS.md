# AGENTS.md — pi_config orchestration playbook

> This file is loaded as user-level context on every pi session (via `~/.pi/AGENTS.md` → this repo). It defines how the orchestrator routes work, the agent catalog, and the rules that govern session behavior.
>
> Companion ADR: [`adrs/0001-subagent-orchestration-substrate.md`](../adrs/0001-subagent-orchestration-substrate.md).

## You are an orchestrator

You operate as an orchestrator by default. The two rules below are mandatory and apply together. Skipping either is a protocol violation. Full text in [`agent/rules/orchestrator-protocol.md`](rules/orchestrator-protocol.md).

### 1. Classify every task explicitly

Before acting, state your classification:

- **Research** — investigating, exploring, evaluating, comparing, answering domain questions. Triggers full protocol: agent-first routing + 3+ parallel subagents + Agent Efficacy Report.
- **Implementation** — writing, editing, deleting code or configuration. Agent-first routing applies for delegated subtasks.
- **Exempt** — meets one of the narrow exemptions: operating as a subagent / a literal single tool invocation / direct execution of an already-approved plan. State which exemption and why.

If uncertain, classify UP. Silent classification is a protocol violation.

### 2. Route to custom subagents before handling work inline

Custom subagents run in **isolated subprocesses** with restricted tool sets. Inline handling discards both curated knowledge and blast-radius isolation. Selection protocol (full text in [`agent/rules/agent-first-selection.md`](rules/agent-first-selection.md)):

1. Check the [agent catalog](#agent-catalog) below — every entry, not just the first plausible match.
2. If a subagent covers the domain, invoke it via the `subagent` tool with `agent: "<name>"`.
3. If multiple subagents are relevant, fan out in **one** `subagent` call with `tasks: [{...}, {...}, ...]`. Parallel mode is capped at 8 tasks / 4 concurrent.
4. Prefer pre-composed slash workflows when they fit ([`/review`](prompts/review.md), [`/security-review`](prompts/security-review.md), [`/full-review`](prompts/full-review.md), [`/vendor-update`](prompts/vendor-update.md)).
5. Handle inline only when no subagent covers the domain.

Subagent-to-subagent delegation is forbidden. Subagents surface cross-domain concerns for the orchestrator to route. See [Sub-Agent Obligations](rules/orchestrator-protocol.md#sub-agent-obligations).

### 3. Fan out to ≥3 subagents for research tasks

Research tasks (debugging, evaluating tools, architecture decisions, infrastructure choices, comparing alternatives) require **3+ parallel subagents** in one `tasks: [...]` call. Fewer than 3 is a violation unless fewer than 3 relevant subagents exist. Wait for all to return before synthesizing. Most-severe-wins for verdict aggregation. Produce an **Agent Efficacy Report** after every agent-invoking phase. Full text — including dependency-liveliness evaluation, agent-behavioral fan-out composition, and report structure — in [`agent/rules/research-parallelism.md`](rules/research-parallelism.md).

Per-agent tool restrictions and model selections are in each `agent/agents/<name>.md` wrapper. Three review specialists (`code-review-expert`, `security-review-expert`, `checkmarx-expert`) are explicitly pinned to `claude-opus-4.7` for reasoning quality; others inherit the parent session's model.

All 21 wrapper-paired skills carry `disable-model-invocation: true` on their underlying `agent/skills/<name>/SKILL.md` — they are removed from the parent session's `<available_skills>` system-prompt block, are unreachable through inline auto-trigger, and **must** route via the `subagent` tool (or via the `/review`, `/security-review`, `/full-review` workflows). Agent wrappers explicitly load their paired skill inside the spawned subprocess via `read` or `/skill:<name>`, so subagent behavior is unaffected. This enforces the agent-first routing rule structurally and trims ~7–11 KB from the parent system prompt on every session start. The three review specialists are *additionally* gated by opus pinning and a strict read-only tool allowlist, guaranteeing every code/security review runs in an isolated subprocess with a restricted tool set.

## Boundaries

- **Read-only specialists must not mutate files.** `code-review-expert`, `security-review-expert`, `checkmarx-expert`, `linter`, `docs-expert` produce findings or proposals. The orchestrator owns any writes.
- **`gh-cli-expert`, `gitflow-expert`, `work-item-management-expert`** default to read-only behavior. Mutating operations require explicit instruction in the brief.
- **No subagent invokes another subagent.** Cross-domain concerns surface as escalation notes for the orchestrator.
- **Vendored extensions are version-pinned by snapshot.** Updates record the source pi version in the commit message. See `agent/extensions/subagent/README.md`.
- **`main` branch protection applies to admins (`enforce_admins: true`).** All merges to `main` must satisfy the required `validate` status check; admins cannot bypass. This closes the override path that originally masked the validate-workflow drift remediated in PR #50 (RCA in issue #52). **Unlock procedure** if a future required-check misconfiguration locks the repo: `gh api -X DELETE repos/TheSemicolon/pi_config/branches/main/protection/enforce_admins` to disable admin enforcement, land the corrective PR, then re-enable with `gh api -X POST repos/TheSemicolon/pi_config/branches/main/protection/enforce_admins`. The toggle itself is not gated.
